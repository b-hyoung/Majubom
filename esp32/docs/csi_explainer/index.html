2 로우데이터 해부 — CSI_DATA 한 줄의 정체

측정 시 시리얼로 1패킷마다 아래 한 줄이 들어온다. 콤마로 나뉜 25개 필드 + 끝의 CSI 배열. 노랑 = 분석에 직접 쓰는 값, 파랑 = 핵심(서브캐리어 I/Q 배열).

배열은 짝수=I(실부), 홀수=Q(허부) → I + jQ 로 묶어 64개 복소 서브캐리어가 된다. 이게 "가슴이 움직이며 WiFi 신호가 어떻게 변했나"의 원본. 진폭 |I+jQ| 과 위상 ∠(I+jQ) 을 따로 쓴다.
※ vital_csi3.parse() 는 실제로 [18] 시각 과 [24] CSI 배열 만 사용 (나머지는 메타/품질 참고).

3 위상(phase) & 진폭(amplitude) — 같은 신호, 다른 관점

서브캐리어 한 개 값 I + jQ 는 복소수다. 벡터(화살표) 로 보면 — 길이=진폭, 각도=위상.
        Q (허부)
        │      ● (I, Q)
        │     ╱
        │    ╱   ← 화살표 길이 = 진폭 |I+jQ| = √(I²+Q²)   "얼마나 세게 도착했나"
        │   ╱θ   ← 화살표 각도 = 위상 ∠(I+jQ) = atan2(Q, I)  "얼마나 밀려서 도착했나"
        └────────── I (실부)
WiFi 파동은 TX → 가슴 반사 → RX 로 온다. 가슴이 움직이면 경로 길이가 바뀌고, 그게 진폭·위상에 다르게 찍힌다.

📶 진폭 = "세기" → 호흡

  1. 여러 경로 파동의 보강/상쇄 간섭 세기
  2. 가슴이 수 mm~cm 크게 움직임 → 진폭 뚜렷이 출렁
  3. 느리고 큰 진동 → 호흡 추출에 적합
  4. 하드웨어 오프셋 영향 적어 안정적

🧭 위상 = "거리/타이밍" → 심박

  1. 경로 길이에 직접 비례: Δφ = 2π·(2·Δd)/λ
  2. 2.4GHz WiFi λ ≈ 12.5cm → 0.5mm 변화도 측정 가능
  3. 심박 진동 0.2~0.5mm 미세 → 진폭엔 안 잡혀도 위상엔 잡힘
  4. 단, 패킷별 위상 오프셋(잡음) 제거 필요 ↓
움직임 크기잘 잡히는 곳주파수
호흡수 mm ~ cm (큼)진폭 |I+jQ|0.1–0.5Hz
심박0.2~0.5 mm (미세)위상 ∠(I+jQ)0.8–2.0Hz
⚠️ 위상의 함정 — TX/RX가 따로 동작해 패킷마다 공통 위상 오프셋(주파수 오차·샘플링 타이밍)이 끼는데, 진짜 심박보다 크다. 그래서 ∠(comp) − 패킷별 평균 으로 오프셋을 빼고 unwrap(±2π 점프 풀기) 한 뒤에야 위상에서 심박이 보인다. 진폭은 이런 오프셋이 없어 호흡이 더 쉽게 나온다.

4 처리 흐름 — 배열에서 심박·호흡까지

복소 CSI를 PCA로 압축한 뒤, 진폭은 호흡(느린 진동), 위상은 심박(빠른 진동)으로 나눠 뽑는다.
공통
I/Q → 복소 CSI (N×64) → 균일 리샘플 → detrend/정규화 → PCA

① 진폭 → 호흡 (resp_rpm)

  1. 진폭 PCA 성분
  2. Welch 스펙트럼
  3. 0.1–0.5Hz(6–30rpm) 최대 피크 → 호흡수
  4. 호흡 envelope 최소창 = 숨참기 구간(발표·검증용)

② 위상 → 심박 (hr_bpm)

  1. 위상 오프셋 제거 + 시간축 unwrap → PCA
  2. 밴드패스 0.8–2.0Hz(48–120bpm)
  3. 자기상관 최대 lag → 심박수
  4. 강도 = ac[lag]/ac[0] → 0.30↑ 이면 reliable

5 서버로 보내는 것 vs 서버가 계산하는 것

CSI는 측정값만 보낸다. 평소값(baseline)·변화량(z-score)·위험단계는 서버가 누적 데이터로 계산.
{
  "timestamp": "2026-06-16T00:00:00Z",  // 측정 시각
  "bed_id": "bed_01",                     // 어느 환자
  "sensor": "csi",
  "raw":     { "hr_bpm":78, "resp_rpm":16, "autocorr_strength":0.41 },  // ② ③ 결과
  "quality": { "reliable":true, "samples_count":8800, "duration_sec":90 },
  "presence":{ "count":1, "confidence":null, "gate_active":true }   // 인원수(CNN)
}
// 서버가 계산: baseline(μ·σ) → z=(x−μ)/σ → total_abs → alert_level

6 지금 겪는 문제점

측정/알고리즘의 본질적 한계와, 시스템 통합에서 실제로 부딪힌 버그를 나눠 정리.

⚠️ 연구·측정 한계

  • HRV 절대값(SDNN ms) — 구현·측정은 했으나 부정확: vital_csi2(진폭 peak)로 SDNN/RMSSD를 실제로 뽑았지만, 애플워치 33~35ms 대비 CSI 66~96ms (2~3배 부풀림) · HR도 ~85bpm 고정. 숨참기 최상조건도 SDNN 159ms → 단일안테나 학술적 한계 확정 → "추세 z-score"로 전환
  • 자기상관 강도 0.30 경계 — 일상 환경(전체신호)에선 0.35처럼 신뢰선에 근접
  • HR z-score 환경 의존 — 통제 ±2bpm vs 일상 5~18bpm 차이
  • 베이스라인 부족 → false positive 폭주 — 4건이면 σ가 좁아 오탐 → 30일 학습 필요
  • 운영 vs 발표 HR 차이 — 운영=전체신호, 발표=숨참기 구간(±2bpm 검증용)

🐞 통합 시 실제 겪은 버그

  • /csi 형식 충돌 — 서버가 옛 형식(amp_mean) 기대 vs 새 형식(raw/quality) → 서버 수정으로 해결
  • 서버 IP·포트 변경 — .48:5001 → .22:5003 으로 바뀌어 송신부 주소 stale
  • 분석 환경 — venv에 scipy 없음 / 서버 flask 미설치 → 실행 불가
  • git submodule 깨짐 — esp-csi/esp-idf 의 .git 포인터가 빈 .git/modules 가리켜 모든 git 명령 실패
